# BuildX

## 名字由来

**BuildX** — 在 OneDev 一站式 DevOps 产品形态之上，用 Go 重新锻造后端，面向 AI-first 研发时代。

研发流程中，需求、代码、评审、构建、发布、运维并非孤立环节，而是像神经网络一样彼此依赖、实时传递上下文。BuildX 的使命，是成为这些环节之间的连接点：让信息无缝流动，让 AI 能够感知完整的研发上下文，让人与机器在同一套流程中协作。

## 核心信念

### 1. 一体化优于拼凑

每个研发环节都与其他环节紧密相关。独立产品 + 脆弱集成 ≠ 真正的效率。一个原生统一的系统，在数据模型、权限、通知、搜索、AI 上下文等方面具有结构性优势。

### 2. AI-first 需要原生衔接

AI 提效的上限，很大程度上取决于它能获取多少跨环节的上下文。当 issue、PR、build、commit 在同一系统内原生关联时，agent 可以：

- 从 issue 直接 checkout 分支并提交 PR
- 从 build 失败自动定位 code comment 和最近变更
- 从 PR review 触发 CI 并关联 issue 状态

这正是 [tod](references/tod) 在 OneDev 生态中验证过的模式 — BuildX 将其提升为平台一等公民。

### 3. Go 是这类系统的理想载体

- **性能**：Git 协议、CI 日志流、并发构建调度
- **部署**：单二进制、小镜像、K8s 友好
- **开放**：无 Oracle JDK 授权困扰，工具链完全自由
- **云原生**：与 container/K8s executor 天然契合

### 4. 站在巨人肩上

[OneDev](references/onedev) 已经证明了「一站式 DevOps」的产品形态是可行的、受欢迎的。BuildX 继承其产品智慧（尤其是前端 UX），用 Go 重新锻造后端，面向 AI-first 时代重新设计衔接层。

本项目**不是** OneDev 的 fork，而是参考其 Java 实现进行 ground-up 的 Go 移植。

## 目标用户

- 希望摆脱多 BG 各自为政、工具割裂的技术治理团队
- 正在推进 AI-first 研发模式转型的组织
- 需要私有化部署、完全开源技术栈的企业
- 个人开发者，希望以一己之力打造完整 DevOps 平台

## 长期愿景

> 让每一个研发动作都在统一的上下文中发生，让 AI 与人类在同一个流程中无缝协作。

这不是又一个 Git 托管或又一个 CI 工具。这是研发流程的操作系统。
